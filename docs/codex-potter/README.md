# Codex Potter 控制面（总说明）

本文档是 `soma-gpu` 的“控制面”总说明，面向主会话与各阶段子代理。它提供一个稳定的、可续跑的协作框架：每轮怎么开始、怎么并行推进、怎么验证、怎么落文档、怎么收尾。

仓库级入口请从：[MAIN.md](../../MAIN.md) 开始。

运行 CodexPotter 时，请把**仓库根 `MAIN.md`** 和 **runtime progress file** 区分开：

- 人类入口：`MAIN.md`
- runtime progress file：`.codexpotter/projects/2026/04/16/1/MAIN.md`
- 当前标准续跑命令：`codex-potter resume 2026/04/16/1 --yolo --rounds <N>`

## 1. 你在这里要解决什么问题

本仓库的 GPU 线路要在“能跑”之上达成更高标准：

- **更快**：把关键路径迁移到 GPU，减少 CPU 往返与 Python 循环，支持 batch 与向量化。
- **更准**：对齐参考实现/基线，数值一致性可衡量、可回归（误差边界与原因可解释）。
- **效果更好**：边界条件与失败模式更少，稳定性与鲁棒性更强。
- **更工程化**：可测试、可复现、可持续迭代，避免“一次性脚本”和“只靠人记忆”的流程。

## 2. 控制面原则（主会话只做调度与轻审阅）

主会话是一台“调度器 + 审阅闸口”，不做长时间单点实现：

- 主会话负责：拆解任务、分派子代理、设置阶段门禁、审阅关键风险与接口、合并与收尾。
- 子代理负责：在其阶段范围内完成具体产物（计划/代码/测试/文档片段）。
- 文档负责：把每轮的输入、决策、产物与未决项固定下来，保证可续跑。

## 3. 每轮固定阶段（强制）

每一轮必须按固定阶段推进，阶段顺序不可跳跃：

1. **计划**：明确目标、范围、验收标准、并行切分与风险。
2. **编码**：子代理分工实现，主会话仅做接口与风险轻审阅。
3. **测试**：把“更快/更准/更工程化”落到可复现的验证命令与指标上。
4. **下一轮建议**：把未完成/风险/后续改进写清楚，作为下一轮输入。
5. **文档总结**：记录本轮决策、变更点、验证结果、已知问题。
6. **提交**：形成可审阅的最小变更集合（必要时分提交）。
7. **结束**：更新运行索引与“上一轮指针”，保证下一轮可从文档启动。

完整协议与模板见：[docs/codex-potter/governance/workflow-protocol.md](./governance/workflow-protocol.md)。

轮次目录与最小交接包规范见：[docs/codex-potter/iterations/README.md](./iterations/README.md)。

## 4. “每轮先读上一轮文档”（强制）

每轮开始前必须先读上一轮的：

- 总结（做了什么，为什么）
- 关键决策（哪些选择被排除，原因是什么）
- 验证结果（跑了什么，结论是什么）
- 未决项与风险（哪些必须延续到下一轮）
- 下一轮建议（下一轮从哪里继续最省力）

> 约定：如果上一轮文档不存在或不完整，本轮的“计划”阶段第一件事就是补齐它，而不是直接开始写代码。

## 5. 子代理与并行（强制：每阶段必须新子代理）

每个阶段必须新开子代理（可以多名并行），禁止复用上一阶段的同名子代理继续工作，原因是：

- 防止阶段门禁被绕过（例如边写代码边“顺手”改测试，导致不可审阅）。
- 强制把输入/输出边界清晰化（阶段产物必须可以被主会话检视与验收）。
- 便于并行（计划/编码/测试可以拆成互不干扰的子任务并行推进）。

## 6. 与 worktree 协作（仓库现状）

仓库当前存在 worktree：`.worktrees/gpu-stageii-foundation/`，聚焦 torch stageii。

推荐协作方式：

- 大改动优先在 worktree 中进行，降低 `main` 上的冲突与上下文负担。
- 控制面文档仍以 `main` 为主入口；实现细节可以在 worktree 中推进，再按阶段协议回到主会话收敛与合并。
