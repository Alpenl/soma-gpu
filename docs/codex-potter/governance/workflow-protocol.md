# 工作流协议（Workflow Protocol）

本协议规定 `soma-gpu` 的控制面工作方式：主会话如何调度，子代理如何分阶段产出，以及每轮如何做到可续跑、可验证、可审阅。

上层入口：

- 仓库级入口：[MAIN.md](../../../MAIN.md)
- 控制面总说明：[docs/codex-potter/README.md](../README.md)
- 轮次索引：[docs/codex-potter/iterations/README.md](../iterations/README.md)

## 1. 定义

- **主会话**：当前对话线程，只做调度与轻审阅，不做长时间单点实现。
- **子代理**：为某个阶段或子任务临时创建的执行单元，按阶段输入输出交付产物。
- **一轮（Round）**：从“读上一轮文档”开始，到“提交与结束”结束的一次完整迭代。
- **阶段门禁**：每阶段必须产出可验收的输出，主会话通过后才能进入下一阶段。

## 2. 硬性规则（必须遵守）

1. **每轮开始必须先读上一轮文档**  
   上一轮的总结/决策/验证/未决项是本轮输入。禁止“凭记忆续写”。
2. **每轮固定阶段顺序不可跳**  
   计划 -> 编码 -> 测试 -> 下一轮建议 -> 文档总结 -> 提交 -> 结束
3. **每个阶段必须新子代理**  
   阶段开始时必须创建新的子代理；不得复用上一阶段的子代理继续推进（可重新开同类型角色，但必须是新的实例）。
4. **尽量并行**  
   能拆分的任务要拆分；计划/编码/测试阶段都应主动寻找并行切分点。
5. **主会话只做调度与轻审阅**  
   主会话不写大段实现；主会话的代码改动应当以“合并、修小洞、修接口”类为主。
6. **入口互链**  
   任何新增的运行记录或关键决策必须回链到 `MAIN.md` 或本协议或控制面 README，避免“孤儿信息”。

## 3. 阶段模板（可复制，确保可续跑）

> 下面模板面向“每一轮”的执行。建议每轮都把“阶段输出”收敛到当轮目录中：`docs/codex-potter/iterations/round-XXXX-<slug>/`。最小交接包应至少包含：`round-overview.md`、`plan.md`、`code.md`、`test.md`、`next-round-suggestions.md`、`summary.md`、`commit.md`、`close.md`。

### 阶段 0：读上一轮文档（强制前置）

**子代理（新建）**：`round-reader`

输入：

- 上一轮运行记录（若不存在，至少读取 `MAIN.md` 的“运行索引”与当前协议）

输出（必须交付给主会话）：

- 上一轮摘要：做了什么、为何如此做
- 未决项列表：必须带优先级与阻塞关系
- 风险与回归点：哪些地方最可能出问题
- “本轮建议起点”：下一步最小可行的切入点

门禁：

- 主会话确认：本轮计划引用了上一轮的未决项与风险点

### 阶段 1：计划（Plan）

**子代理（新建，可并行）**：

- `planner-a`：需求与范围（更快/更准/效果/工程化）拆解为可验收条目
- `planner-b`：代码库探索与切分建议（模块边界、依赖、最小改动面）
- `planner-c`：验证策略（测试、基准、对齐指标、可复现命令）

输出（必须）：

- 目标与非目标（明确“本轮不做什么”）
- 验收标准（至少包含：速度指标、精度/一致性指标、工程化指标之一）
- 并行切分方案（子任务列表，尽量互斥写集合）
- 风险清单与缓解（每条风险对应一个缓解动作）
- 预计改动路径（文件/模块级别，不要求逐行）
- 阶段文档：`plan.md`

门禁：

- 主会话确认：验收标准与验证命令存在，且并行切分不互相踩踏

### 阶段 2：编码（Code）

**子代理（新建，强烈建议并行）**：

- `worker-*`：每人负责一块互斥写集合（例如某个模块/某类算子/某条 pipeline）
- `integrator`：只负责接口收敛与集成（避免所有人同时改同一层 glue 代码）

输出（必须）：

- 可审阅的最小改动集（小步、可解释）
- 关键实现说明（只写“为什么这么做”，不写大段教程）
- 若涉及 GPU 加速：明确数据形状、dtype、device 流转与同步点
- 阶段文档：`code.md`

门禁：

- 主会话轻审阅：接口边界、错误处理、潜在数值风险点已被标注

### 阶段 3：测试（Test）

**子代理（新建，可并行）**：

- `tester-a`：功能正确性与回归（单测/集成测试/对齐误差）
- `tester-b`：性能验证（benchmark，至少一条可复现命令与指标）
- `tester-c`：稳定性与边界（异常输入、空 batch、dtype/device 边界）

输出（必须）：

- 可复现的测试命令（含环境假设）
- 结论（通过/不通过）与证据（输出摘要/指标）
- 失败时的下一步定位建议（最短定位路径）
- 阶段文档：`test.md`

门禁：

- 主会话确认：至少覆盖“更准”或“更快”中的一项硬证据

### 阶段 4：下一轮建议（Next）

**子代理（新建）**：`next-round-advisor`

输出（必须）：

- 未完成项：按“阻塞/重要/可选”分类
- 下一轮最小切入点（1 到 3 条）
- 潜在技术债：未来会反噬的点（越具体越好）
- 阶段文档：`next-round-suggestions.md`

门禁：

- 主会话确认：建议可操作，不是泛泛而谈

### 阶段 5：文档总结（Docs）

**子代理（新建）**：`doc-summarizer`

输出（必须）：

- 本轮总结（变更点、原因、影响）
- 验证摘要（跑了什么、结果如何）
- 已知问题与风险（以及是否接受）
- 更新互链（确保 `MAIN.md` 运行索引可续跑）
- 阶段文档：`summary.md`

门禁：

- 主会话确认：下一轮能仅靠文档继续推进

### 阶段 6：提交（Commit）

**子代理（新建）**：`commit-prepper`

输出（必须）：

- 提交建议：是否需要拆分提交、提交信息草案
- 变更审阅清单：哪些文件是高风险区
- 阶段文档：`commit.md`

门禁：

- 主会话确认：提交粒度与审阅路径合理

### 阶段 7：结束（Close）

**子代理（新建）**：`round-closer`

输出（必须）：

- 更新 `MAIN.md` 的“每轮运行索引”（新增一行）
- 标注“上一轮指针”与“本轮产物指针”
- 阶段文档：`close.md`

门禁：

- 主会话确认：索引可追踪，互链完整

## 4. 并行策略（建议）

并行的基本原则：**写集合互斥 + 接口集中收敛**。

- 在“计划”阶段就声明每个子代理的写集合（模块/文件范围），减少冲突。
- 在“编码”阶段保留一个 `integrator` 角色专门做 glue 层收敛，避免多人同时改入口代码。
- 在“测试”阶段拆成正确性/性能/边界三条并行线，避免把测试当成单线程收尾。

## 5. 与 worktree 的协作约定（当前仓库现状）

仓库存在 worktree：`.worktrees/gpu-stageii-foundation/`，聚焦 torch stageii。

约定：

- 该 worktree 用于承载 torch stageii 相关的大改动与实验；其变更最终需要回到主会话按阶段门禁收敛。
- `main` 尽量保持干净，控制面文档与最终合并以可审阅为第一优先级。

## 6. 典型失败模式（用协议去避免）

- 跳过“读上一轮文档”：导致重复劳动与错误决策复现。
- 计划缺少验收标准：最后只能“感觉更快/更准”，不可验证。
- 编码阶段与测试阶段混在一起：不可审阅，回退困难。
- 文档不互链：信息散落，下一轮无法续跑。
