# GPU 优化迭代统一指标框架（Metrics Framework）

上层入口：

- 仓库级入口：[MAIN.md](../../../MAIN.md)
- 控制面总说明：[docs/codex-potter/README.md](../README.md)
- 工作流协议：[docs/codex-potter/governance/workflow-protocol.md](workflow-protocol.md)
- 轮次索引：[docs/codex-potter/iterations/README.md](../iterations/README.md)

本文档用于规范后续所有 GPU 优化轮次（iterations / rounds）的指标口径、测量方法、输出格式与决策门槛。目标是让每一轮的“提速/降耗/提质”都能被**可复现地度量**，并能明确回答：

- 变快了多少？（速度）
- 偏差/误差是否可接受？（误差）
- 主观观感有没有变差或变好？（视觉效果）
- 是否可维护、可上线、可回滚？（工程化）

## 1. 基本原则

- 同一轮次内，所有对比必须使用同一套输入、同一台机器或同一类机器、同一套环境参数。
- 任何“提升”都必须提供：基线（baseline） + 方案（candidate） + 复现信息（repro）。
- 默认以“端到端（E2E）”为主指标，同时保留“核/算子（kernel/op）”的解释性指标。
- 回归预算（regression budget）必须显式声明：允许牺牲什么、不允许牺牲什么。
- 指标分级：P0（阻断上线/合入）与 P1（跟踪但不阻断）。

## 2. 评测对象与术语

- **基线 Baseline**：当前主分支（或当前发布基线）的实现。
- **候选 Candidate**：本轮准备评估的实现（可以来自分支、patch 或 `.worktrees/*` 的候选资产）。
- **参考 Reference**：用于计算误差/视觉差异的“高精度”或“已验证正确”的实现/结果（可为 CPU、较慢 GPU 路径、或已冻结的版本输出）。
- **场景/工作负载 Workload**：固定的一组输入样本与运行参数（分辨率、batch、步数、采样器、模型版本等）。

## 3. 指标总览（四大类）

每轮必须至少覆盖四类指标中的 P0 指标，并在 round 文档中给出统一表格。

### 3.1 速度（Speed）

P0（必须）：

- **E2E 延迟**：`latency_ms`（每次任务/每帧/每 batch 的耗时，说明口径：p50/p90/p99）。
- **吞吐**：`throughput`（FPS / it/s / samples/s，说明口径：steady-state）。

P1（建议）：

- **分段耗时**：前处理/后处理/IO/推理/渲染等阶段耗时（有助定位瓶颈）。
- **GPU Kernel 时间占比**：用于解释为何 E2E 变化。
- **冷启动时间**：首次运行或首次加载模型/资源的耗时。

速度测量约定：

- 必须报告 warm-up 策略（例如预热 N 次，丢弃前 N 次）。
- 必须报告统计方式（至少 30 次采样或固定时长采样）。
- 必须报告波动（标准差或置信区间），避免偶然误差。

### 3.2 误差（Accuracy / Error）

P0（必须）：

- **数值差异**：与 Reference 的差异度量，至少一种（按业务选）：`L1/L2`、`PSNR`、`SSIM`、`max_abs_diff`、或任务指标（如 IoU、Top1）。
- **稳定性**：是否出现 NaN/Inf、溢出、崩溃、明显非确定性（需说明 seed/确定性配置）。

P1（建议）：

- **长尾误差**：p99 / max case 的误差分布。
- **误差热区定位**：差异图、层/算子级别的误差定位（如有工具链）。

误差验收约定（示例，具体由每轮决策声明）：

- “默认不允许误差指标退化超过 X% / 或绝对阈值超过 Y。”
- 若引入近似算法：必须写清楚“误差换速度”的业务可接受范围，并给出样例对比。

### 3.3 视觉效果（Visual Quality）

视觉效果的特点是主观与客观并存。建议使用“客观指标 + 代表性可视化样例 + 人工审阅”三件套。

P0（必须）：

- **样例对比产物**：至少提供固定样例集的对比图（baseline vs candidate vs diff）或对比视频。
- **关键伪影检查**：是否出现明显伪影（闪烁、条纹、锯齿、色偏、banding、ghosting、噪声结构变化等），必须写结论。

P1（建议）：

- **客观视觉指标**：`SSIM/PSNR/LPIPS`（按业务选择）。
- **主观评分**：小样本 MOS（至少 2 人复核，记录结论与分歧）。

视觉验收约定：

- 如果“视觉变差”但能接受：必须明确写“接受理由”和“适用范围”，并制定回滚开关或降级路径。

### 3.4 工程化（Engineering）

P0（必须）：

- **可复现性**：构建方式、运行命令、依赖版本、硬件/驱动信息完整。
- **可回滚**：提供回滚策略（编译开关、运行时 flag、或单 PR revert 的粒度）。
- **可维护性底线**：新增依赖/复杂度必须解释；不可把临时 hack 直接入主干。

P1（建议）：

- **可移植性**：不同 GPU 架构/不同驱动版本的兼容性说明。
- **资源占用**：显存峰值 `vram_peak_mb`、显存碎片化风险、CPU 占用、系统内存占用。
- **可观测性**：日志、统计、profiling hook 是否齐全。

工程验收约定：

- 任何引入新依赖、启用非默认编译选项、或使用未验证的内核路径，都必须写出风险与 fallback。

## 4. 统一输出格式（Scorecard）

每轮的 round 文档必须包含下面的评分卡（可直接复制）：

```text
硬件/环境：
- GPU: <型号> / 驱动: <版本> / CUDA: <版本> / 时钟策略: <锁频/默认> / 电源策略: <说明>
- OS: <版本> / 编译器: <版本> / 关键依赖: <版本>

Workload：
- 数据集/样例集：<路径或生成方式>（固定版本）
- 关键参数：<分辨率/batch/steps/...>
- warm-up：<策略>
- 采样：<次数或时长>，统计：p50/p90/p99 + mean/std

对比结论（Baseline -> Candidate）：
- 速度：
  - latency_ms: p50 <x> -> <y> (Δ <...>)
  - throughput: <x> -> <y> (Δ <...>)
- 误差：
  - <metric>: <x> -> <y> (阈值：<...>) 结论：<通过/不通过/需决策>
  - 稳定性：<无 NaN/Inf | 有，说明>
- 视觉：
  - 样例集：<链接/路径> 结论：<通过/不通过/需决策>
- 工程化：
  - 可复现：<通过/不通过，缺失项>
  - 回滚：<有/无，说明>

决策：
- 结论：<合入/不合入/合入但默认关闭/继续试验>
- 决断点：<写明触发条件与负责人>
```

## 5. 复现与可信度要求（Repro Requirements）

每轮至少要能做到“同一台机器上 24 小时内可复现”：

- 固定版本：代码 commit、模型/资源版本、数据样例版本。
- 固定命令：一条或少数几条命令能跑完基线与候选，并生成同口径结果。
- 固定输出：产出原始测量日志（原始数据）与汇总表（scorecard）。

## 6. 将 `.worktrees/gpu-stageii-foundation` 纳入评估的统一口径

`.worktrees/gpu-stageii-foundation` 视为“候选实现资产（candidate asset）”，允许被用于：

- 直接作为 Candidate 与 Baseline 做 A/B 指标对比；
- 作为“可 cherry-pick 的变更集合”，逐步引入并分阶段评估；
- 作为工程化基座（构建、profiling、可观测性、测试）供后续轮次复用。

纳入评估时，必须在 round 文档里补充：

- 资产描述：该 worktree 解决什么问题，包含哪些关键变更。
- 评估策略：全量对比 or 分批 cherry-pick；每批的验收门槛。
- 风险清单：与主干差异、依赖变化、API 破坏、性能波动来源。
- 决策出口：采纳（merge/cherry-pick）/部分采纳/不采纳，并说明理由与后续动作。
