## 1. 问题背景与现状

### 1.1 仓库与工作区并存的现实约束

当前代码环境同时存在：

- 主仓库工作区：`/home/alpen/DEV/soma-gpu`
- 并存 worktree：`/home/alpen/DEV/soma-gpu/.worktrees/gpu-stageii-foundation`（以及可能的后续其他 worktree）

这意味着我们必须把“做决策/调度/记录”的控制面和“执行/改代码/跑命令”的执行面明确分离，否则会出现：

- 变更落点不明确：同一任务可能误改到主仓库或误改到某个 worktree。
- 上下文混淆：讨论的是 A 工作区的状态，但命令却在 B 工作区执行。
- 回滚/续跑困难：缺少明确的检查点与状态记录，导致失败后无法可靠恢复。

### 1.2 控制面要解决的核心问题

控制面不是“再造一个框架”，而是把协作流程显式化、可复用化，确保：

- 每一轮工作“要做什么、为什么做、在哪里做、做到了什么程度”都可追踪。
- 任务可拆解、可并行，但最终能稳定收敛到可合并、可验证的结果。
- 遇到失败或中断时能恢复（resume）而不是重来（restart）。

## 2. 为什么采用“主会话调度 + 子代理执行 + 文档驱动”

### 2.1 主会话调度（Control Plane in Main Session）

主会话负责：

- 统一的目标与约束裁决：范围、优先级、是否允许改动 GPU 核心路径等。
- 任务拆分与派发：把可并行/可隔离的工作拆给子代理或不同 worktree。
- 质量门禁：收敛到“可验证、可解释、可复现”的提交前状态（即使此轮不提交，也要形成可复用的 plan 入口）。
- 文档作为真实来源（source of truth）：用文档承载状态、计划、风险与决策，而不是散落在聊天里。

理由：主会话天然拥有“全局视角”和“最终合并权”，适合作为控制面。

### 2.2 子代理执行（Execution Plane via Sub-agents / Worktrees）

子代理负责：

- 在明确的 ownership 范围内执行具体工作：探测、实现、修复、实验、跑测试、生成数据等。
- 在隔离的工作区内操作：优先在目标 worktree 执行，避免污染主仓库。
- 产出可审阅的证据：命令记录、日志、结果文件、可重复的步骤。

理由：执行工作天然可并行且可隔离；把执行与决策拆开能降低主会话负担并减少误操作。

### 2.3 文档驱动（Docs as Interface）

文档是控制面与执行面的“接口”，用于：

- 定义输入：目标、边界、验收标准、依赖、风险。
- 定义输出：完成情况、证据链接、可复现命令、已知问题。
- 定义状态：每一轮的状态机位置、下一步、阻塞点、恢复点。

理由：文档是可版本化、可 review、可 diff 的；对“跨轮次续跑”尤其关键。

## 3. 文档体系设计（Information Architecture）

本项目的控制面文档体系需要满足两个目标：

- 对人：一眼知道当前在做什么、进度如何、为什么这么做。
- 对后续计划生成：能直接作为 implementation plan 的输入（机器可读程度不强求，但结构必须稳定）。

建议的文档类型（本 spec 只定义结构与原则，不强制一次性落地所有文件）：

- `docs/codex-potter/specs/`：规格说明（Spec）。描述“是什么/为什么/边界/机制”。
- `docs/codex-potter/plans/`：实现计划（Plan）。描述“怎么做/按什么顺序/验收与风险控制”。
- `docs/codex-potter/runbooks/`：运行手册（Runbook）。描述“如何执行命令/如何排障/如何复现”。
- `docs/codex-potter/logs/`：轮次日志（Round Log）。描述“这一轮做了什么/证据/结论/下一步”。
- `docs/codex-potter/checkpoints/`：检查点（Checkpoint）。记录“可恢复的状态摘要”，例如：目标分支、worktree 位置、关键开关、基线指标。

文档命名与组织原则：

- 以日期为主键：便于按轮次串联与续跑（例如 `YYYY-MM-DD-<topic>.md`）。
- 以“控制面关键对象”为索引：轮次、状态、指标、恢复点、约束。
- 所有关键决策必须落到 spec/plan 中，避免只存在于对话。

## 4. 轮次状态机（Round State Machine）

### 4.1 状态定义

每一轮（Round）以“从输入到可追踪产出”为单位，建议使用如下状态机（可按实际裁剪，但语义要保持一致）：

1. `draft`：目标与范围尚未冻结；收集信息、澄清约束。
2. `scoped`：范围冻结；明确 worktree/ownership；明确不做事项（non-goals）。
3. `planned`：形成可执行的 plan（步骤、验收、风险、回滚/恢复点）。
4. `dispatched`：任务派发给子代理/执行面；明确输入与预期输出。
5. `executing`：执行中；持续产出证据与中间结果。
6. `verifying`：验证中；跑测试/基准/一致性校验；收敛到“证据足够”。
7. `integrating`：整合中；将可合并内容汇聚（或将结论固化到文档）。
8. `completed`：本轮完成；产出轮次总结与下一轮入口。
9. `blocked`：受阻；明确阻塞原因、所需输入、恢复策略。
10. `aborted`：终止；说明终止原因与清理策略（尽量不做 destructive 回滚）。

### 4.2 状态迁移规则（原则）

- 只能在 `scoped` 之后进行派发（`dispatched`），避免边做边改目标导致浪费。
- 任何“可能改变结论”的执行都必须进入 `verifying`，否则不允许宣布完成。
- 允许从任意状态进入 `blocked`；解除阻塞后回到原状态或 `planned`（取决于阻塞是否改变计划）。
- `aborted` 不是失败的同义词：它是“明确停止并留下可复用痕迹”的状态。

### 4.3 轮次产物（每轮至少具备）

- 范围声明：goals / non-goals / constraints（尤其是“是否触碰 GPU 核心路径”）。
- 工作区声明：在哪个 worktree 执行、是否允许改主仓库。
- 验收声明：本轮“完成”的客观条件是什么。
- 证据与可复现步骤：关键命令、关键输出位置、关键对比指标。
- 恢复点：失败后从哪里继续（而不是从头再来）。

## 5. 指标设计原则（Metrics）

控制面指标不是为了“好看”，而是为了驱动可验证的工程收敛。原则如下：

### 5.1 先定义“正确性”，再谈“性能”

- 任何性能指标（吞吐、延迟、显存占用）都必须建立在正确性不退化的前提上。
- 正确性指标应优先可自动化：单元测试、集成测试、golden output、回归样例等。

### 5.2 指标必须可复现、可比较

- 明确运行环境：GPU 型号、驱动版本、CUDA 版本、关键依赖版本。
- 明确数据与随机性：固定 seed、记录数据版本或样例哈希。
- 明确测量方法：warmup 次数、统计口径（mean/p95）、采样窗口。

### 5.3 指标分层：控制面指标 vs 系统指标

控制面指标（用于衡量“协作与收敛能力”）示例：

- 轮次收敛率：`completed / (completed + aborted)`（按周期统计）。
- 续跑成功率：从 `blocked` 或中断恢复后能否继续推进。
- 变更定位清晰度：每个任务是否都有明确 worktree 与 ownership（可用审阅 checklist 打分）。

系统指标（用于衡量“GPU/模型/系统行为”）示例（本轮不强制落地，只定义原则）：

- 训练/推理吞吐、延迟、显存峰值、OOM 频率、数值稳定性等。

### 5.4 指标要能驱动决策

- 每个指标必须关联一个决策：若指标退化/提升，下一步怎么做。
- 避免收集不可行动指标（例如“看起来更快”但无法复现或无对照）。

## 6. 恢复与续跑原则（Recovery / Resume）

### 6.1 恢复优先于回滚

工程实践上更可靠的是“可恢复（resume）”，而不是“强行回滚（rollback）”：

- 回滚依赖人为判断且容易误删他人工作。
- 恢复通过检查点与幂等步骤可自动化推进。

### 6.2 幂等与检查点

控制面约束：

- 所有关键步骤必须可幂等执行：重复执行不会把状态推向不可控。
- 每轮至少一个“检查点”描述：分支/commit、worktree、关键配置、基线指标。

执行面约束：

- 产出物必须可定位：日志路径、结果文件名、关键输出摘要。
- 失败要留下“可诊断证据”，而不是只留下“失败了”。

### 6.3 中断后的续跑策略

当出现中断（如命令失败、环境异常、会话结束）时：

1. 优先从最近检查点恢复上下文：确认 worktree、分支、未完成步骤。
2. 只重跑必要步骤：重跑的前提是有明确的“为什么要重跑”。
3. 若计划不再成立：回到 `planned` 更新 plan，并记录变更原因。

## 7. 为何此轮只做初始化，不做 GPU 核心大改

本轮目标是“控制面初始化与固化”，不进入 GPU 核心路径的大改动，原因：

- 当前存在多工作区并存与 ownership 约束：先把工作流与落点规则固化，避免在高风险代码上发生误改。
- GPU 核心改动的风险面大：涉及性能、数值稳定性、依赖环境与硬件差异；在缺少基线指标与恢复机制时推进会导致不可控。
- 文档体系与状态机尚未落地：先把“如何做事”的机制稳定下来，后续才能安全地做“做什么”的大变更。
- 本轮产物应服务于下一轮的 implementation plan：把范围、指标、恢复策略、文档结构先定义清楚，降低后续实施成本。

本轮明确 non-goals（不做事项）：

- 不进行大规模 GPU kernel 重写/算子替换/并行策略重构。
- 不承诺性能提升数字（除非已建立可复现基线与对照）。
- 不引入会影响全局架构的大依赖迁移（例如大版本 CUDA/框架升级）。

## 8. 后续 implementation plan 的输入约束（本 spec 输出格式要求）

为确保本 spec 可直接喂给后续 plan 生成与执行，本文件需要被视为“约束源”，后续 plan 应遵守：

- 计划必须显式声明 round 状态机起点与目标终点（例如：`planned -> verifying`）。
- 计划必须写清 worktree 与 ownership（哪个目录、哪些文件允许改）。
- 计划必须定义至少一个验证门（tests/bench/golden）与一个恢复点（checkpoint）。
- 计划不得违反本轮 non-goals；如需突破必须先更新 spec 并记录理由。
